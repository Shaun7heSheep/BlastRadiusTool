# v1.1 UI Updates ‚Äî Implementation Plan

## Context

The Blast Radius Tool's UI (Blazor WASM + 3d-force-graph) is functionally complete but visually minimal ‚Äî colored spheres on a single 3D view with no interactivity beyond real-time blast radius highlights. Version 1.1 adds three features: a 2D/3D graph toggle, Azure service icons on nodes, and a side panel for filtering by application. These changes are scoped entirely to the frontend (`BlastRadiusUI/`); no backend changes are required.

The `feature/ui-v2` branch has one commit so far (color palette refresh in `graph.js`). All v1.1 work builds on top of that.

---

## Key Architecture Decisions

1. **Single `graph.js` file** ‚Äî grows from ~190 to ~400 lines. Splitting would complicate the `IJSObjectReference` import pattern in Blazor. One import, one module, same exported function surface.

2. **Destroy-and-recreate on 2D/3D toggle** ‚Äî keeping two live graph instances doubles memory and creates state-sync problems. The toggle is infrequent; sub-second recreation is acceptable.

3. **Unified `_applyVisualState()` function** ‚Äî three visual concerns (default colors, blast radius highlights, app filter dimming) must compose without clobbering each other. One internal function reads all module-scoped state and sets all graph visual callbacks in one pass. `highlightBlastRadius`, `resetHighlights`, `highlightApp`, and `clearAppFilter` become thin wrappers that store state then call `_applyVisualState()`.

4. **`force-graph` (2D) is acceptable** ‚Äî same author (vasturiano), identical API surface, same dependency tree minus Three.js. Update invariant 9 in CLAUDE.md to note this exception.

5. **ResizeObserver replaces window resize listener** ‚Äî handles both window resize and side panel open/close (which changes container width without triggering `window.resize`).

## Phase 1: 2D/3D Graph Toggle

> **Agent:** `frontend`
>
> **Scope:** Add `force-graph` 2D library, refactor `graph.js` for dual-mode rendering, add toggle buttons in Home.razor.

### Step 1.1 ‚Äî Add `force-graph` 2D library

**File:** `BlastRadiusUI/wwwroot/index.html`

- Add `<script src="https://unpkg.com/force-graph@1.49.5/dist/force-graph.min.js"></script>` before the existing 3d-force-graph script tag
- Both UMD globals (`ForceGraph` for 2D, `ForceGraph3D` for 3D) coexist without conflict

### Step 1.2 ‚Äî Refactor `graph.js` for dual-mode rendering

**File:** `BlastRadiusUI/wwwroot/js/graph.js`

New module-scoped state:
- `_mode` ("3d" | "2d") ‚Äî current rendering mode
- `_currentData` ‚Äî stashed `graphData` for re-init after toggle
- `_currentResult` ‚Äî stashed blast radius result (already implicitly needed)
- `_appFilter` ‚Äî `{ appName, nodeIds }` or `null` (for Phase 3, add the slot now)
- `_container` ‚Äî cached container element reference

Changes to existing exports:
- `initGraph(elementId, graphData, mode)` ‚Äî accept a `mode` parameter (default `"3d"`). Branch on mode: `ForceGraph3D()(container)` vs `ForceGraph()(container)`. Common config (nodeId, nodeLabel, linkSource, linkTarget, arrows, particles, background) is shared. Stash `_currentData`, `_mode`, `_container`.
- `highlightBlastRadius(result)` ‚Äî stash into `_currentResult`, call `_applyVisualState()`
- `resetHighlights()` ‚Äî clear `_currentResult`, call `_applyVisualState()`
- `disposeGraph()` ‚Äî handle both 2D cleanup (no `_destructor`, just null the reference) and 3D cleanup

New export:
- `toggleMode(newMode)` ‚Äî if `newMode === _mode`, no-op. Otherwise: dispose current graph, call `initGraph` with stored `_currentData` and new mode, re-apply `_currentResult` if present, re-apply `_appFilter` if present.

New internal:
- `_applyVisualState()` ‚Äî reads `_currentResult` and `_appFilter`, sets all node/link visual callbacks in one pass. Priority: blast radius > app filter > defaults.

### Step 1.3 ‚Äî Add toggle UI to Home.razor

**File:** `BlastRadiusUI/Pages/Home.razor`

- Add `private string _graphMode = "3d";` state field
- Add toggle buttons in the header bar (right side, before the status badge): two `FluentButton` components styled as a segmented button group ‚Äî "3D" (default active) and "2D"
- Handler `SetGraphMode(string mode)`: sets `_graphMode`, calls `_jsModule.InvokeVoidAsync("toggleMode", mode)`
- Update `OnAfterRenderAsync` to pass `_graphMode` to `initGraph`

### Step 1.4 ‚Äî CSS for toggle buttons

**File:** `BlastRadiusUI/wwwroot/css/app.css`

- Add `.graph-mode-toggle` button group styles (flex, gap, border radius matching header)

## Phase 2: Azure Service Icons on Nodes

> **Agent:** `frontend`
>
> **Scope:** Preload the 8 SVG icons already in `wwwroot/icons/`, render them as Three.js sprites (3D) and canvas images (2D), add blast-radius halo effects on icons.

### Step 2.1 ‚Äî Icon preloading infrastructure

**File:** `BlastRadiusUI/wwwroot/js/graph.js`

- New `_iconCache = {}` module-scoped dictionary
- New async helper `_preloadIcons(nodes)`: extracts unique `azureType` values, loads each as an `Image` from `/icons/${azureType}.svg`, stores in `_iconCache[azureType]`. Returns a Promise that resolves when all images are loaded. Includes error handling ‚Äî if an SVG fails to load, the cache entry is `null` and the node falls back to a colored shape.
- Call `await _preloadIcons(graphData.nodes)` at the top of `initGraph` before constructing the graph. This means `initGraph` becomes `async` ‚Äî update Blazor interop call accordingly (it already uses `InvokeVoidAsync` which handles async JS).

### Step 2.2 ‚Äî 3D node rendering with icons

**File:** `BlastRadiusUI/wwwroot/js/graph.js`

- In the 3D branch of `initGraph`: set `.nodeThreeObject(node => _createSpriteForNode(node))` and `.nodeThreeObjectExtend(false)` to replace default spheres
- New helper `_createSpriteForNode(node)`: creates a 64√ó64 off-screen canvas, draws the cached SVG image (or a colored circle fallback), creates a `THREE.CanvasTexture`, creates a `THREE.Sprite` with `THREE.SpriteMaterial({ map: texture, transparent: true })`, scales to appropriate size based on criticality
- `THREE` is available as a global from the 3d-force-graph bundle (it re-exports Three.js)

### Step 2.3 ‚Äî 2D node rendering with icons

**File:** `BlastRadiusUI/wwwroot/js/graph.js`

- In the 2D branch of `initGraph`: set `.nodeCanvasObject((node, ctx, globalScale) => _drawNodeIcon2D(node, ctx, globalScale))` and `.nodeCanvasObjectMode(() => "replace")`
- New helper `_drawNodeIcon2D(node, ctx, globalScale)`: draws the cached SVG image centered on the node position via `ctx.drawImage`. Draws a label below the icon. Falls back to a colored circle if icon not cached.

### Step 2.4 ‚Äî Blast radius visual states on icons

**File:** `BlastRadiusUI/wwwroot/js/graph.js`

Update `_applyVisualState()` and the node rendering helpers:

- **Failed node (amber):** Draw a pulsing amber ring/halo around the icon ‚Äî 2D: `ctx.arc()` + `ctx.stroke()` with `#F59E0B`; 3D: composite a glowing border onto the sprite canvas
- **Affected node (red):** Draw a red ring ‚Äî same technique with `#EF4444`
- **Dimmed node (app filter, not selected):** Reduce icon opacity ‚Äî 2D: `ctx.globalAlpha = 0.25`; 3D: adjust sprite material opacity
- **Normal node:** Full opacity, type-colored background circle behind the icon for visual consistency

The key insight: in 2D mode, `nodeCanvasObject` is called every frame, so reading module-scoped `_currentResult` and `_appFilter` state inside the callback is both natural and performant. In 3D mode, sprites are static objects ‚Äî `_applyVisualState()` must re-create sprites (call `_graph.nodeThreeObject(...)` again to trigger re-render) when blast or filter state changes.

**Verification:** Visual ‚Äî render the graph in both modes, confirm icons display. Trigger a blast radius, confirm amber/red halos appear instead of sphere color changes.

## Phase 3: Side Panel with Application Filter

> **Agent:** `frontend` (component + Home.razor integration + JS interop + CSS)
>
> **Agent:** `tester` (unit tests for app filter logic)
>
> **Scope:** New Fluent UI side panel listing the 5 app groups from the graph data. Selecting one highlights that app's nodes and edges while dimming everything else. Blast radius highlights always take priority over filtering.

### Step 3.1 ‚Äî Create the AppFilterPanel component

> Agent: `frontend`

**New file:** `BlastRadiusUI/Components/AppFilterPanel.razor`

- Parameters: `List<string> Apps`, `string? SelectedApp`, `EventCallback<string?> SelectedAppChanged`, `bool IsOpen`, `EventCallback<bool> IsOpenChanged`
- Renders a vertical list using `FluentListbox` with `FluentOption` items
- First option: "All Applications" (clears filter)
- One `FluentOption` per app name (sorted alphabetically)
- Panel header with title "Applications" and a dismiss/close `FluentButton`
- Panel slides in from the left, overlaying or pushing the graph

### Step 3.2 ‚Äî Compute app names from graph data

> Agent: `frontend`

**File:** `BlastRadiusUI/Pages/Home.razor`

- New state: `private List<string> _appNames = new();`, `private string? _selectedApp;`, `private bool _filterPanelOpen;`
- After `_graphData` is fetched in `OnInitializedAsync`, compute: `_appNames = _graphData.Nodes.Select(n => n.App).Distinct().OrderBy(a => a).ToList();`
- Add a filter `FluentButton` (funnel icon) in the header bar that toggles `_filterPanelOpen`

### Step 3.3 ‚Äî Layout integration

> Agent: `frontend`

**File:** `BlastRadiusUI/Pages/Home.razor`

- Wrap the `#graph-container` div in a `.graph-area` flex row
- Add `<AppFilterPanel>` component inside `.graph-area`, before the graph container
- Panel width: 260px when open, 0 when closed (CSS transition)
- Graph container fills remaining space via `flex: 1`

### Step 3.4 ‚Äî JS interop for app highlighting

> Agent: `frontend`

**File:** `BlastRadiusUI/wwwroot/js/graph.js`

New exports:
- `highlightApp(appName, nodeIds)` ‚Äî stores `_appFilter = { appName, nodeIds: new Set(nodeIds) }`, calls `_applyVisualState()`
- `clearAppFilter()` ‚Äî sets `_appFilter = null`, calls `_applyVisualState()`

`_applyVisualState()` composition rules:
1. If a node is the failed node ‚Üí amber halo, full opacity (blast wins)
2. If a node is affected ‚Üí red halo, full opacity (blast wins)
3. If app filter is active AND node is NOT in the filtered set AND node is NOT blast-affected ‚Üí dim (opacity 0.25, muted color)
4. If app filter is active AND node IS in the filtered set ‚Üí full opacity, normal color
5. Links: dim links where neither endpoint is in the filtered set (unless blast-affected)

### Step 3.5 ‚Äî Blazor event handler

> Agent: `frontend`

**File:** `BlastRadiusUI/Pages/Home.razor`

- `OnAppFilterChanged(string? appName)` method:
  - If `appName` is null or "All": call `_jsModule.InvokeVoidAsync("clearAppFilter")`
  - Else: compute `nodeIds = _graphData.Nodes.Where(n => n.App == appName).Select(n => n.Id).ToList()`, call `_jsModule.InvokeVoidAsync("highlightApp", appName, nodeIds)`

### Step 3.6 ‚Äî Replace window resize with ResizeObserver

> Agent: `frontend`

**File:** `BlastRadiusUI/wwwroot/js/graph.js`

- Replace `window.addEventListener("resize", _resizeHandler)` with a `ResizeObserver` on the container element
- This correctly handles both window resize AND side panel open/close (which changes container width without firing window resize)
- Store as `_resizeObserver`, disconnect in `disposeGraph()`

### Step 3.7 ‚Äî CSS for side panel

> Agent: `frontend`

**File:** `BlastRadiusUI/wwwroot/css/app.css`

- `.app-filter-panel` ‚Äî fixed width 260px, dark background matching header, vertical layout, overflow-y auto
- `.app-filter-panel--closed` ‚Äî width 0, overflow hidden, with CSS transition
- `.app-filter-header` ‚Äî flex row with title and close button
- `.graph-area` ‚Äî flex row, flex:1, min-height:0, overflow:hidden

### Step 3.8 ‚Äî Tests for app filter logic

> Agent: `tester`

**New file:** `BlastRadiusUI.Tests/AppFilterTests.cs`

- `ExtractAppNames_ReturnsDistinctSorted` ‚Äî given a GraphData with nodes across multiple apps, verify the extraction logic produces sorted, deduplicated app names
- `ExtractAppNames_EmptyGraph_ReturnsEmptyList`
- `FilterNodesByApp_ReturnsCorrectIds` ‚Äî given app "orders", verify only order-function and order-db IDs returned
- `FilterNodesByApp_SharedApp_ReturnsAllSharedServices` ‚Äî "shared" app has 5 nodes, verify all returned
- `FilterNodesByApp_UnknownApp_ReturnsEmpty`

These tests verify the C# filtering logic that will live in Home.razor (or extracted to a static helper on the model). They do NOT test JS interop.
